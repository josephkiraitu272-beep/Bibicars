# Бізнес-логіка BIBI Cars

Цей документ описує **логіку процесів**: як працює «пригон» від ліда
до видачі ключів, хто за що відповідає та як рухаються дані.

---

## 1. Ключова ідея

Клієнт хоче пригнати авто з аукціону США/Кореї. Платформа проводить його
через весь ланцюг: підбір авто → розрахунок вартості → депозит → ставка на
аукціоні → виграш → контракт → оплата → логістика → митниця → видача.
Кожен етап відображається в CRM (для персоналу) і в кабінеті клієнта.

---

## 2. Життєвий цикл угоди (поетапно)

```
[Лід] → [Кваліфікація] → [Депозит] → [Ставка/Виграш] → [Контракт]
      → [Оплата] → [Доставка: океан+суша] → [Митниця] → [Видача]
```

**Етап 1 — Лід.**
Лід приходить із вітрини (форма, калькулятор, дзвінок). Система фіксує
джерело (UTM), призначає менеджера, виставляє пріоритет і SLA.
- Сервіси: `lead_priority`, `lead_sla`, `lead_notifications`, `utm_propagation`.
- Воркери: `lead_sla`, `lead_reminders`, `escalations_wakeup`.

**Етап 2 — Кваліфікація та дзвінки.**
Менеджер працює з лідом: дзвінки, задачі, нагадування. Кожен дзвінок
отримує евристичний скоринг (намір, рівень інтересу, рекомендований
наступний крок) на основі тривалості та історії контактів.

**Етап 3 — Депозит.**
Клієнт вносить депозит — сигнал готовності до ставки. Створюється
інвойс, фіксується платіж.

**Етап 4 — Ставка/Виграш.**
Обраний лот додається у watchlist; воркер `watchlist_live_poll` стежить за
живою ціною/статусом. Після виграшу угода переходить у стадію оформлення.

**Етап 5 — Контракт.**
Юридичний воркфлоу (`legal_workflow.py`): генерація документів зі шаблонів,
надсилання клієнту на e-підпис (кабінет → «Sign Contract»).

**Етап 6 — Оплата.**
Інвойси та платежі (`payments_tracking.py`); опційно Stripe. Воркер
`payment_reminder` нагадує про неоплачені інвойси.

**Етап 7 — Доставка.**
Логістика: океанська частина (трекінг судна через VesselFinder) +
внутрішня. Сервіси `shipments`, `delivery_health`; воркер `tracking_worker`.
Події пишуться в `shipment_events` і відображаються на таймлайні клієнта.

**Етап 8 — Митниця та видача.**
Фінальні документи, розмитнення, закриття угоди, видача ключів.

## 3. Ролі та їхня зона відповідальності

### Адмін (`/admin`)
Повний контроль: Control Panel, вся CRM, Operations 360, Forecasting 360,
Contract 360, Finance, Delivery 360, калькулятор, персонал, парсери,
налаштування, KPI, аудит, безпека. Адмін бачить «master dashboard».

### Тімлід (`/team/dashboard`)
Операційний контроль команди: Manager Load Board (навантаження,
скори, проблемні ліди), переназначення лідів (reassignments),
нагляд за SLA, неоплачені інвойси, проблеми доставки. Це «зріз»
master dashboard в межах команди.

### Менеджер (`/manager`)
Особистий простір (My Workspace): Health-Score, гарячі ліди, задачі,
дзвінки, інвойси, замовлення, відвантаження, wishlist, Top Deals.

### Клієнт (`/cabinet/:id`)
Особистий кабінет: статус авто (прогрес 5 етапів), підписання
контрактів, інвойси, доставка, VIN-трекінг, Carfax, обране (favorites),
порівняння, нотатки, контакт свого менеджера. Оновлення — у реальному
часі (таймлайн подій).

## 4. Потік даних про авто

```
Джерела (bidmotors, lemon, westmotors, …)
        │  парсери
        ▼
   Сирі дані по лотах/VIN  ──→  enrichment_worker  ──→  vin_data / vin_data_lemon
        │                                                  │
        ▼                                                  ▼
  multisource_resolver / resolver_engine  ──→  єдина картка авто в каталозі
```

- **Парсинг** — збір сирих даних (ціна, фото, характеристики, лот).
- **Збагачення** — додавання історії VIN, фото, додаткових атрибутів.
- **Резолвінг** — злиття даних про одне авто з різних джерел
  (найкраще значення по кожному полю) в єдину картку.

---

## 5. Сповіщення та SLA

- Правила сповіщень — `notification_rules`; доставка in-app + email outbox.
- SLA по лідах — воркер `lead_sla` періодично перевіряє прострочення
  і піднімає ескалації.

---

## 6. Скоринг дзвінків

Ендпойнт `POST /api/ai/call-analysis/{call_id}` робить евристичний аналіз
дзвінка на основі тривалості та історії контактів (намір, рівень інтересу,
рекомендований outcome). Зовнішній LLM НЕ потрібен.
